# PDF Export Foundation

**Status:** Implemented  
**Date:** 2026-07-23  
**Scope:** Presentation-only PDF adapter for the existing Export Foundation

## Architecture Integration

PDF is a third adapter behind the existing Export Registry. It introduces no new request, authorization, query, filter, document, filename, guard-rail, or HTTP path.

```text
existing centralized Admin / Operator authorization
  → existing Export Coordinator
  → existing Reporting Read Service
  → existing presentation-neutral Export Document
  → existing Export Registry
      ├─ CSV adapter
      ├─ Excel adapter
      └─ PDF adapter
  → existing private no-store download response
```

`src/exports/pdf-adapter.ts` imports only PDFKit and Export types. It does not import Prisma, Reporting readers or filters, authorization, Dashboard, or Transaction Engine code. Reporting and the existing document builders finish all selection, ownership scope, summaries, calculations, dates, Rupiah values, labels, and ordering before the adapter receives the Export Document.

The Export Coordinator and `ReportExportDocument` / `ReportExportAdapter` contracts remain unchanged.

## Adapter Responsibilities

The PDF adapter performs presentation only:

- creates a minimal A4 landscape document;
- renders the document title, generated time, period, and existing summary items;
- derives headers exclusively from `document.columns`;
- renders every `document.rows` value in declared column and row order;
- proportionally sizes columns from display-ready content;
- wraps cell text, continues a row across pages when necessary, and avoids dropping oversized cell content;
- repeats the table header on every table page;
- adds `Halaman X dari Y` numbering; and
- returns one buffered PDF byte array.

The adapter sanitizes PDF control characters and maps the display-only mathematical minus glyph to an ASCII minus because PDFKit's built-in Helvetica encoding does not contain that glyph. It does not parse or recalculate the amount.

## Registry and HTTP Integration

The supported formats are:

| Format | Adapter | Media type | UI |
|---|---|---|---|
| CSV | `csvExportAdapter` | `text/csv; charset=utf-8` | Exposed |
| Excel (`xlsx`) | `excelExportAdapter` | `application/vnd.openxmlformats-officedocument.spreadsheetml.sheet` | Exposed |
| PDF | `pdfExportAdapter` | `application/pdf` | Exposed |

The existing Admin and Operator endpoints accept `format=csv`, `format=xlsx`, or `format=pdf`. The UI derives download actions from `exportRegistry.availableFormats()`, so registration exposes **Unduh PDF** without page-specific format logic. The Coordinator continues to select the adapter and reuse the existing privacy-safe Jakarta filename generator for `.pdf`.

## Pagination and Guard Rails

PDF uses the existing `EXPORT_MAX_ROWS` preflight and optional `EXPORT_MAX_BYTES` estimate/final-byte checks. Oversized reports are rejected from the first authorized Reporting page before later pages or PDF rendering. The Coordinator still gathers accepted Reporting pages sequentially and supplies one complete Export Document.

PDFKit produces stream chunks, but the adapter intentionally buffers and combines them because the established adapter contract returns `Uint8Array` and the HTTP layer performs an exact final byte-limit check. PDF is therefore bounded synchronous output, not a streaming or high-volume guarantee. Per-format deployment capacity, deadlines, concurrency, and cross-page read consistency remain Production Hardening work.

## Dependency Decision

The adapter uses PDFKit `0.19.1`, a maintained Node PDF generation library with text measurement, explicit page control, document metadata, and table support. The implementation uses explicit layout primitives to guarantee repeated headers and continued wrapped rows under the project's Export Document contract.

`pdfjs-dist` is test-only and round-trips generated bytes to verify page count, metadata, and rendered text. It is not part of the production export path.

## Verification Coverage

Regression tests cover:

- PDF adapter registration and registry selection;
- `%PDF-` generation, MIME type, and `.pdf` filename generation;
- document title metadata, generated time, period, summaries, headers, and transaction rows;
- Coordinator integration with every Reporting page and existing guard rails;
- multi-page generation, repeated headers, page numbering, and final-row preservation;
- existing HTTP attachment behavior and registry-derived UI exposure; and
- existing CSV, Excel, ownership-isolation, Admin privacy, authorization, and Export Contract behavior.

## Explicitly Deferred

- Logos, organization branding, signatures, watermarks, cover pages, charts, graphs, custom themes, advanced typography, and premium layouts.
- Tagged-PDF accessibility refinement and custom font embedding.
- Streaming, asynchronous generation, background jobs, queues, object storage, snapshots, and read-model redesign.

These changes belong to Export Presentation Polish or Production Hardening and must continue to consume the same Export Document.
